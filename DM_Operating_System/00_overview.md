# DM Operating System (DM-OS) — Overview

> **Purpose:** A portable personal playbook for delivery management. Synthesized from professional standards (Step 1) and honest self-assessment (Step 2). This is not theory — it is the documented method for how delivery management is actually practiced, with gaps filled in with explicit approaches to adopt.
>
> **Confidence test:** When fully internalized, it should be possible to walk into any company that hires a DM and know exactly what to do in Week 1, how to run each core function, and how to adapt when the context differs.

## The Portable Confidence Standard

Before entering any new delivery environment, the following should be true:

- [ ] Week 1 checklist is ready to execute
- [ ] RAID log template is ready to populate
- [ ] Requirements template includes acceptance criteria and priority fields
- [ ] Communication matrix template is ready to fill in
- [ ] Change log is ready from Day 1
- [ ] Escalation thresholds will be defined in the first stakeholder meeting
- [ ] Go/no-go checklist is on hand

The instinct that exists is real. This document makes it visible, repeatable, and transferable.

## Cross-Cutting Patterns to Watch

Three structural tendencies that cut across all functions and are the root cause of most gaps:

| Pattern | Risk | Antidote |
|---|---|---|
| **Documentation Deficit** | Everything lives in memory — resets with every new environment | One written artifact per function, even if minimal |
| **Product Company Bubble** | Financial/commercial layer is entirely absent | Deliberate study of Function 6 before any outsourcing engagement |
| **"Solve It Myself First" Reflex** | Bad news delayed; changes self-approved; escalation timing slow | Apply the 24h rule consistently across communication and escalation |

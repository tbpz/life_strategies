# Function 9: Change Control
*How to handle scope changes, timeline shifts, and surprises*

**Principle:** Change is inevitable. Unmanaged change is how projects fail. The mechanism that converts surprises into decisions — and decisions into updated plans — is change control.

---

## Method

*At Project Start:*
- Define how changes are submitted (via PM, directly to DM, via ticket system?)
- State explicitly: no change is implemented before impact is assessed and confirmed in writing

*For Every Change Request:*
1. Assess impact on: scope, timeline, budget, resources, risk
2. Identify who has approval authority (client, PM, steering committee, or DM within defined threshold)
3. Obtain written confirmation from the authority before any execution begins
4. Write a two-line record: what changed, who approved it, date approved
5. Communicate the change and its impact to all affected parties — team and stakeholders

*The Two-Line Change Record (minimum documentation):*
> *Change: [description]. Approved by: [name]. Date: [date]. Impact: [timeline/scope effect].*

This habit alone closes the documentation gap that caused problems at Zalo.

*Cumulative Change Tracking:*
- Maintain a running list of all accepted changes
- At each milestone, review the list: what is the combined effect of all changes on scope, timeline, and effort?
- Individual changes look small. The list makes the cumulative impact visible before it doubles the project.

*Change vs. Defect:*
- **Change request** = work not in the original scope → requires approval and (in outsourcing) may trigger a change order
- **Defect** = work that should have been delivered correctly in scope → requires resolution, not approval

---

## Cadence

Change record created for every accepted change. Cumulative list reviewed at each milestone.

---

## Artifact

Change log (running list: description, approver, date, impact). Two-line change record per item.

---

## Adaptation Notes

| Context | Adjustment |
|---|---|
| Fixed-price outsourcing | Change control is financial protection; every out-of-scope request = change order trigger |
| Agile / product | Changes enter backlog through refinement; prioritization is the governance mechanism |
| US/EU enterprise client | Formal change request form + paper trail often contractually required |
| High-frequency change environment | Batch changes into weekly review to avoid constant plan disruption |

---

## Self-Check

- Is there a written record for every change accepted in the last sprint/month?
- Has the cumulative impact of all changes been reviewed recently?
- Has any change been implemented without explicit written confirmation from the appropriate authority?

---

## Gap to Close

The two-line change record is the minimum viable habit. Start applying it immediately — even informally in a chat message or email reply.

# Function 4: Stakeholder Communication
*Who gets what information, how often, and in what format*

**Principle:** Delivery confidence is built or destroyed through communication, not just outcomes. Stakeholders who feel informed are forgiving of problems. Stakeholders who feel surprised become adversarial — even when results are good.

---

## Method

*At Project Start — Stakeholder Map:*
For each key stakeholder, document:
- What they care about (outcomes, risks, schedule, budget?)
- Their preferred communication channel (async, meeting, email, Slack)
- How much detail they want (summary/headline vs. full breakdown)
- Their sensitivity level (high-anxiety = needs more frequent contact)

*Communication Matrix:*
A simple table defining: who gets what, how often, in what format.

| Stakeholder | What They Receive | Frequency | Format |
|---|---|---|---|
| PM | Full project status: progress, blockers, risks, decisions needed | Weekly | Written sync or meeting |
| Tech Lead | Execution detail: blockers, dependencies, upcoming work | As needed | Direct conversation or chat |
| Line Manager / CTO | Summary: risk status, critical issues, escalations | Weekly sync | Verbal + follow-up note if critical |
| Client (if applicable) | External status: progress, upcoming milestones, any flags | As per contract | Written report |

*Status Update Structure (consistent format every time):*
1. Progress since last update
2. Upcoming work this period
3. Risks/issues (with status)
4. Decisions needed from them

*Bad News Rule:*
Bad news travels before it is fully solved. The threshold: if a problem will still exist in 24 hours regardless of individual effort, it gets communicated now — not after attempting to solve it independently. The framing includes the problem, the impact, and what is being done — not just the problem.

---

## Cadence

Stakeholder map created at project start. Communication matrix defined at project start. Status updates on the defined cadence — stakeholders should never need to ask.

---

## Artifact

Stakeholder map (table), communication matrix (table), recurring status update (structured format).

---

## Adaptation Notes

| Context | Adjustment |
|---|---|
| US/EU client | Async-first; written summary before video calls; Loom for complex updates |
| Executive stakeholder | One-pager max; RAG status; decisions needed highlighted |
| High-anxiety stakeholder | Over-communicate temporarily to rebuild trust, then normalize |

---

## Self-Check

- Is there a written stakeholder map for this project?
- When did the last status update go out — and did the stakeholder initiate it, or was it sent proactively?
- Has any bad news been held back because it wasn't fully solved yet?

---

## Gap to Close

Define a personal rule for communication cadence that defaults to "DM sends before stakeholder asks." Even if the update is brief, the habit of initiating preserves trust.

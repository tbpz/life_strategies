# Work Audit Experiment — Diagnosing the Inefficiency Feeling

> **Status:** Data collection. One working week. ~2 minutes/day, tagged at end of day.
> **Purpose:** Convert a *feeling* of inefficiency into *evidence*, then let the evidence — not the feeling — name the problem.
> This file is self-contained. It re-establishes the full context on its own, so the experiment can resume in a fresh session.

---

## Background — Why This Experiment Exists

A persistent feeling reports: "too much time goes to busy, unimportant operational work, leaving no room for the work that actually builds capability." The session that produced this experiment reached three conclusions that reframe that feeling:

1. **There is no separate "important" vs. "unimportant" work competing for time.** There is one stream — operational delivery work — run in one of two modes:
   - **Manual mode:** acting personally as the follow-up, the router, the safety net, the reviewer. Regenerates endlessly; does not compound.
   - **Leveraged mode:** a system, an owner, or a forcing-function carries the handoff. Shrinks itself over time; *is* the capability being built.
2. **"Unimportant" is the wrong axis and the reason the search keeps coming up empty.** The operational work is genuinely important — it ships projects. Hunting for "unimportant work to cut" finds almost nothing. The correct axis is: **did this need ME specifically (judgment / authority / relationship), or does it merely route through me by default?**
3. **The feeling is not yet evidence.** The conviction "there must be waste" was reached *before* looking. For the anxiety pattern documented in `career_strategy.md`, a sense of "busy and inefficient" is a baseline hum, independent of actual efficiency. This experiment exists to test the conviction honestly — including the possibility that the work is already efficient and the feeling is the pattern, not a fact.

---

## What To Capture

For one working week, log every meaningful work chunk — one line each. Granularity: roughly anything that took 15+ minutes or interrupted focus. Tag at end of day; do not interrupt the workday to log.

### Daily Log

| Date | Work chunk (short) | Me-test | Origin (system-gap?) | Recurrence |
|---|---|---|---|---|
| _ex: 06-27_ | _Reviewed dev research, framed issue, pulled tech lead in_ | _Y_ | _N_ | _Rec_ |
| _ex: 06-27_ | _Chased QA to confirm she could start testing the closed dev task_ | _N_ | _Y_ | _Rec_ |
| | | | | |

*(Replace the example rows. Add rows freely; keep entries short — the value is the tags, not prose.)*

### Tag Definitions

- **Me-test** — Did this need *this person specifically* — their judgment, authority, or relationship?
  - `Y` = genuinely required them (a real decision, an escalation only they can make, a relationship only they hold).
  - `N` = important work that merely routed through them; a system, template, owner, or clear rule could have carried it.
- **Origin (system-gap?)** — Did this work *exist only because an earlier step lacked a system*? (i.e., reactive work generated by a prior un-systematized handoff.)
  - `Y` = regenerated reactive work. `N` = work that would exist regardless.
- **Recurrence** — Does this *kind* of work recur, or was it a one-off?
  - `Rec` = recurring pattern. `One` = genuinely one-time.

**The target set is the rows tagged `N / Rec`** (and especially `N / Y / Rec`): important, recurring work that did not need this person and was generated by a missing system. That is the provable, sized leverage opportunity — as opposed to a feeling.

---

## Pre-Committed Verdict (set before the data exists, so the goalpost cannot move)

- **If clear recurring `N` patterns appear** (one or more distinct types of important, non-me-specific work that recur): a **real leverage problem** is confirmed. Response: systematize those patterns one at a time, mapped to the relevant DM-OS function, until each function's Self-Check passes on evidence. (See `DM_Operating_System/00_overview.md`.)
- **If most rows are `Y`** (the work genuinely needed this person) and `N / Rec` rows are scarce or non-recurring: the inefficiency feeling is the **anxiety pattern, not a workload fact.** Response: close the loop. Do **not** hunt harder for hidden waste. Carry the feeling as a feeling.

Both outcomes are wins. The only way to fail the experiment is to refuse the second verdict when the data supports it.

---

## End-of-Week Diagnostic Questions

1. Of all logged chunks, what share is `N` (routed through, did not need this person)?
2. Among the `N` rows, which *types* recur? Name each recurring type concretely.
3. For each recurring `N` type: which DM-OS function would absorb it if its system were actually running (RAID, acceptance criteria, onboarding checklist, change record, QA go/no-go)?
4. Did any `N / Rec` work, left manual, generate *further* downstream work? (Trace one chain.)
5. Honest read: does the data support a leverage problem, or the anxiety-pattern verdict?
